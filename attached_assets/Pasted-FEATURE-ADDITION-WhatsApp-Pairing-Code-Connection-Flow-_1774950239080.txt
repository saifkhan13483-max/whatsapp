SECTION 1: FEATURE OVERVIEW
═══════════════════════════════════════════════════════════════════

This feature allows users to link their WhatsApp account to WaTracker using
WhatsApp's official "Link with Phone Number" (Pairing Code) method — the same
mechanism used by WhatsApp Web/Desktop. No QR code scan is needed. Instead:

1. User opens the "Connect WhatsApp" flow inside WaTracker
2. WaTracker backend requests a pairing code from WhatsApp's servers
   by submitting the user's phone number
3. WhatsApp generates an 8-character alphanumeric one-time pairing code
4. WaTracker displays this code prominently on screen
5. User opens their WhatsApp app → Settings → Linked Devices →
   Link a Device → "Link with phone number instead"
6. User enters the 8-character code shown in WaTracker
7. WhatsApp confirms the pairing — WaTracker backend receives a
   valid session/token and stores it securely
8. WaTracker is now registered as a companion/linked device and can
   monitor online status changes, contacts, and activity

This uses WhatsApp's official Multi-Device (MD) architecture.
The backend should use the Baileys library (WhiskeySockets/Baileys)
or a compatible WhatsApp Web reverse-engineered client that supports
pairing code linking.

SECTION 2: NEW BACKEND API ENDPOINTS (add to existing Express backend)
═══════════════════════════════════════════════════════════════════

All endpoints are authenticated (require valid JWT cookie).

POST /api/whatsapp/request-pairing-code
  Body: { phoneNumber: string }  — E.164 format e.g. "+923001234567"
  Action: Initializes a Baileys WhatsApp client session for this user,
          requests a pairing code for the given phone number
  Response: { pairingCode: string, expiresAt: string (ISO timestamp) }
  Error cases:
    - 400: invalid phone number format
    - 409: WhatsApp already linked for this user
    - 429: too many requests (rate limit: max 3 attempts per 10 minutes)
    - 500: WhatsApp server error

GET /api/whatsapp/connection-status
  Response: {
    status: 'not_connected' | 'pending_pairing' | 'connected' | 'disconnected',
    phoneNumber?: string,       — masked e.g. "+92 300 ***4567"
    connectedAt?: string,       — ISO timestamp
    pairingCode?: string,       — only present when status = 'pending_pairing'
    pairingCodeExpiresAt?: string
  }

POST /api/whatsapp/disconnect
  Action: Logs out the WhatsApp linked device session, clears stored credentials
  Response: { success: true }

GET /api/whatsapp/pairing-code-status
  Action: Polls whether the pairing code has been accepted by WhatsApp
  Response: {
    accepted: boolean,
    status: 'waiting' | 'accepted' | 'expired' | 'error'
  }

POST /api/whatsapp/reconnect
  Action: Attempts to restore a previously linked session from stored credentials
  Response: { status: 'connected' | 'failed' }

SECTION 7: useWhatsAppConnection HOOK
═══════════════════════════════════════════════════════════════════

File: hooks/useWhatsAppConnection.ts

Responsibilities:
- Query: GET /api/whatsapp/connection-status (staleTime: 30000, refetchOnFocus: true)
- Mutation: requestPairingCode(phoneNumber: string) — POST /api/whatsapp/request-pairing-code
  On success: invalidates connection-status query, returns { pairingCode, expiresAt }
  On error: throws with parsed error message
- Query: pollPairingStatus() — GET /api/whatsapp/pairing-code-status
  refetchInterval: controlled externally (3000 when polling is active, false otherwise)
  Only enabled when connection status is 'pending_pairing'
- Mutation: disconnect() — POST /api/whatsapp/disconnect
  On success: invalidates connection-status query, shows success toast
  On error: shows error toast
- Mutation: reconnect() — POST /api/whatsapp/reconnect
  On success: invalidates connection-status query

Exports:
  connectionStatus        — data from connection-status query
  isLoadingStatus         — loading state
  pairingCodeStatus       — data from poll query
  requestPairingCode      — mutation function
  isRequestingCode        — loading state for request mutation
  disconnect              — mutation function
  isDisconnecting         — loading state
  reconnect               — mutation function
  isReconnecting          — loading state
  isPolling               — boolean state, set true on Step 2, false on Step 3 or expiry
  setIsPolling            — setter

All 404 responses from NEW endpoints must be caught and returned as:
{ status: 'not_connected' } for connection-status
{ accepted: false, status: 'waiting' } for pairing-code-status
This ensures the UI degrades gracefully if backend isn't fully built yet.

SECTION 8: BACKEND IMPLEMENTATION NOTES
═══════════════════════════════════════════════════════════════════

Add to existing Express + TypeScript + Drizzle ORM backend (port 3001).
Do NOT rebuild or restart the server — add new routes to existing router structure.

Database additions (new Drizzle schema table):
whatsapp_sessions table:
  id              serial primary key
  userId          integer references users(id) on delete cascade
  phoneNumber     varchar(20) — E.164 format, encrypted at rest
  maskedPhone     varchar(20) — e.g. "+92 300 ***4567"
  status          varchar(20) — 'pending_pairing' | 'connected' | 'disconnected'
  pairingCode     varchar(10) — current pairing code
  pairingCodeExpiresAt  timestamp
  sessionData     text        — encrypted Baileys session JSON (sensitive)
  connectedAt     timestamp
  lastActiveAt    timestamp
  createdAt       timestamp default now()
  updatedAt       timestamp default now()

Baileys Integration:
- Install: @whiskeysockets/baileys
- Use makeWASocket with { auth: state, printQRInTerminal: false }
- For pairing code: call sock.requestPairingCode(phoneNumber) after socket initialization
- Store session credentials using Drizzle (encrypt sessionData field using crypto with APP_SECRET env var)
- Each user gets their own isolated Baileys socket instance
- Store active socket instances in a Map<userId, WASocket> in memory
- On server restart: attempt to restore all connected sessions from database

Session lifecycle:
- POST /api/whatsapp/request-pairing-code:
  1. Validate phone number (libphonenumber-js or simple regex)
  2. Check rate limit (max 3 per 10 min per user, store attempts in memory or Redis)
  3. Initialize Baileys socket for this user
  4. Call sock.requestPairingCode(phoneNumber)
  5. Listen for 'connection.update' event
  6. Store pairing code + expiry (5 minutes) in whatsapp_sessions table
  7. Return pairing code to frontend

- GET /api/whatsapp/pairing-code-status:
  1. Check whatsapp_sessions for this user
  2. If connection.update fired with isNewLogin = true → status = 'accepted'
  3. If pairingCodeExpiresAt < now → status = 'expired'
  4. Otherwise → status = 'waiting'

- On successful pairing (Baileys 'connection.update' with connection = 'open'):
  1. Update whatsapp_sessions: status = 'connected', connectedAt = now()
  2. Save Baileys creds to sessionData (encrypted)
  3. Emit real-time update via Socket.IO or Server-Sent Events (optional, if already set up)

- On disconnection:
  1. Update status = 'disconnected'
  2. Attempt auto-reconnect up to 3 times with 5s delay
  3. After 3 failures, mark as 'disconnected' and notify user

Error handling:
- If Baileys throws "invalid phone number": return 400
- If WhatsApp rate-limits the pairing request: return 429 with retry-after header
- All Baileys errors should be caught and mapped to appropriate HTTP status codes

SECTION 10: UI/UX RULES FOR THIS FEATURE
═══════════════════════════════════════════════════════════════════

- The 8-character pairing code must be the visual focal point of Step 2 —
  large, bold, clearly readable, impossible to miss
- The countdown timer must always be visible when the code is active
- Instructions must be simple numbered steps — no paragraphs
- Never expose the raw phone number in any UI — always show masked version
  after Step 1 is submitted

  (users should never feel trapped)

  check connection-status on screen focus — if already connected, jump to Step 3
- Polling MUST stop when: screen is unmounted, step changes to 3, code expires
  Use useFocusEffect from expo-router to start/stop polling based on screen focus
- All error messages must be human-readable, not technical:
  - "Could not reach WhatsApp. Please check your internet connection."
  - "This phone number doesn't appear to be registered on WhatsApp."
  - "Too many attempts. Please wait 10 minutes and try again."
  - "The code has expired. Please request a new one."
